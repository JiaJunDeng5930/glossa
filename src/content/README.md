# Content Rendering Notes

The content script renders glosses in the page text flow after translation is active for the page. Activation starts from `settings.autoTranslateEnabled` or the action popup message `glossa.activateTranslation`; `settings.translateShortcutKey` toggles the page-local translation state. Enabling opens a `gloss.session` port, sends `gloss.scan.start`, streams `gloss.scan.chunk` messages with backpressure, then sends `gloss.scan.end`. Background acknowledges chunks with `gloss.chunk.ack` and streams `gloss.token` outcomes through the same port. Disabling clears any queued scan, closes gloss ports, unwraps rendered tokens, and leaves future mutation or scroll rescans idle until translation is enabled again. `createGlossOverlay` keeps the `#glossa-overlay` host for extension identity, settings variables, and the selection veil, then wraps each rendered source word with an inline `data-glossa-token` span.

Each token wrapper keeps the original word on the page text baseline and positions the gloss label above it. The label and source word share the same vertical centerline. The wrapper reserves the wider of the label and source word, so dense labels create uneven word spacing through normal browser layout instead of covering neighboring labels or page text.

Rendering is outcome-driven. `pending` creates the wrapper with a spinner label, `ready` creates or updates the wrapper with the translated label, `hidden` unwraps the token, and `error` marks the label with the error badge. The overlay keeps a segment table for original text nodes so streamed outcomes in the same text node can still apply from the original offsets after an earlier token has already been wrapped. Stable token ids include root kind, parent path, lemma, source text, sentence offset, and a local source fingerprint.

Selection mode is held by `SelectionController`. While the configured shortcut is down, it sets the page selection flag, `createGlossOverlay.setSelectionMode(true)` shows a fixed gray transparent veil, and page interaction is frozen by consuming pointer preparation, scroll, and shortcut key events. A keydown outside the configured shortcut or page focus loss exits selection mode so chorded shortcuts and OS-level window switches leave the page unfrozen. Clicks in this mode are converted into word selections: an existing `data-glossa-token` reuses its wrapper metadata, and plain text creates a manual token when the clicked text offset lands on an English word.

Card feedback uses the same inline wrapper as gloss rendering. Click selection immediately writes `data-glossa-feedback="card-pending"` and shows `...`. A `word.card.duplicate` response opens a top-right confirmation prompt; confirmation resends the click with duplicate approval, while timeout or cancellation clears the pending feedback. A `word.clicked.ok` response counts as successful card creation only when `payload.noteId` is a number. Success writes `data-glossa-feedback="card-success"` and failure writes `data-glossa-feedback="card-error"`. A wrapper with translated text stores that text in `data-glossa-gloss-display`, restores it after pending, and changes background color; a wrapper without translated text displays `✓` or `×` through the same badge renderer.

Mutation handling treats render and cleanup DOM writes as owned mutations. External page mutations invalidate the active scan version, keep old gloss ports available for pending-wrapper reconciliation, prune disconnected wrappers, and schedule a reconciliation scan while the page activation flag is on. Route changes close all gloss ports, call `clear()`, unwrap rendered tokens, and restore the activation flag from `settings.autoTranslateEnabled`.

Pending wrappers can outlive the scan version that created them during ordinary page mutations. Stale token outcomes may update an existing pending wrapper when its stored surface, lemma, offsets, source fingerprint, and local text context still match the page. This lets slow gloss lookups finish after nearby DOM changes while route changes still clear all page-local rendering and ports.

Scan chunking uses count and time boundaries. The producer scans only text ranges that currently intersect the viewport, sends a chunk after 64 candidate tokens or 16ms, and keeps at most 4 unacknowledged chunks in flight. Rendered gloss wrappers are left in place when scrolling moves them outside the viewport; route changes and translation disable still clear wrappers. Large text nodes can split one sentence into repeated sentence entries with token subsets. Token outcomes that arrive while a scan is still walking live text nodes are queued and replayed after the scan finishes; queued outcomes keep their original token mapping when that source text is still valid after a page invalidation. A terminal gloss error aborts the active chunk producer and releases pending chunk acknowledgements.
